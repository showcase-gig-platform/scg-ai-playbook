# Confluence Epic 参加チケット作成

このコマンドを実行すると、Confluence MCP（および必要に応じて Slack MCP / Jira MCP）を使い、**対象スペースのタスクテンプレートに合わせて** Epic 参加用タスクの内容を作成する。入力は 4 パターンに対応する。固定フォーマットは使わず、そのスペースのテンプレート構造に従う。

---

## 入力パターンの判定

ユーザー入力を次のいずれかで判定する。

| 入力の形 | パターン | 次の節 |
|----------|----------|--------|
| `https://*.slack.com/archives/...` のような Slack リンク | Slack 経由 | 入力パターン A: Slack リンク |
| Confluence ページ ID / タイトル＋スペースキー / Confluence URL | Confluence 経由 | 入力パターン B: Confluence |
| Jira のプロジェクト／ボード URL（例: `.../projects/DATA/boards/543`） | Jira 経由 | 入力パターン D: Jira プロジェクト |
| 上記以外のテキスト（Epic 名・概要の貼り付けなど） | 直接テキスト | 入力パターン C: 直接テキスト |

---

## 入力パターン A: Slack リンク

1. **リンクをパースする**  
   Slack のメッセージ URL は `https://{workspace}.slack.com/archives/{channel_id}/p{timestamp}` 形式。`channel_id` と `ts`（タイムスタンプ。`p1234567890123456` の場合は `1234567890.123456` に変換）を抽出する。
2. **Slack MCP で内容取得**  
   - スレッドの親メッセージ: `conversations_history` で channel ID と適宜 `ts` を指定。  
   - スレッド内: `conversations_replies` で channel ID と親の `ts` を指定。  
   取得した本文・スレッドを「Epic の説明」として解釈する。
3. **対象の特定**  
   Slack の内容から Epic 名・スペース名を読み取る。ユーザーが Confluence スペースキーや **Jira プロジェクト／ボード URL** を指定した場合はそれに従う（Jira URL なら 入力パターン D のテンプレートを使う）。不明な場合は「タスクテンプレートを参照したい Confluence のスペースキー、または Jira のプロジェクト／ボード URL」を確認する。
4. **タスク作成**  
   対象が Confluence なら「対象スペースのタスクテンプレート取得」、Jira なら「Jira プロジェクトの課題フォームをテンプレートとする場合」に従い、タスク内容を生成する。

---

## 入力パターン B: Confluence

1. **対象を特定する**  
   - ページ ID のみ → Confluence MCP の `confluence_get_page` に page ID を渡す。  
   - タイトル＋スペースキー → `confluence_get_page` に title と spaceKey を渡す。  
   - Confluence URL の場合は URL から pageId（または space + title）を抽出して上記のいずれかに変換する。
2. **Epic ページと子を取得**  
   - `confluence_get_page` で Epic ページの本文・メタデータを取得。**取得したページの spaceKey を「対象スペース」として記録する。**  
   - 必要なら `confluence_get_page_children` で子ページ一覧を取得し、参加範囲を把握する。
3. **タスク作成**  
   「対象スペースのタスクテンプレート取得」に従い、そのスペースのタスクテンプレートを取得し、テンプレートの形式でタスク内容を生成する。

---

## 入力パターン C: 直接テキスト

1. ユーザーが貼り付けたテキストをそのまま「Epic の説明」として扱う。
2. **対象スペース**がテキストに含まれていればそれを使う。含まれていなければユーザーに「タスクテンプレートを参照したい Confluence のスペースキー（またはスペース名）」を確認する。
3. 「対象スペースのタスクテンプレート取得」に従いテンプレートを取得し、その形式でタスク内容を生成する。

---

## 入力パターン D: Jira プロジェクト

ユーザーが Jira のプロジェクト／ボード URL を指定した場合（例: `https://showcasegig.atlassian.net/jira/software/c/projects/DATA/boards/543`）。

1. **プロジェクトキーを抽出する**  
   URL から `projects/{project_key}` を取得（上記例なら `DATA`）。
2. **テンプレートの参照先**  
   そのプロジェクトの**課題フォーム**がテンプレートになる。Confluence ではなく Jira の「主要な詳細」セクション（依頼内容・依頼の背景 など）に合わせてタスク内容を構成する。
3. **タスク作成**  
   「Jira プロジェクトの課題フォームをテンプレートとする場合」に従い、タスク内容を生成し、必要なら Jira MCP でチケット作成まで行う。

---

## 対象スペースのタスクテンプレート取得

タスクの**見出し・項目・書式**は、対象スペースのタスクテンプレートに合わせる。固定フォーマットで書かない。

1. **テンプレートページの取得**  
   Confluence MCP で対象スペース（spaceKey またはスペース名）内のタスクテンプレートを探す。
   - `confluence_search` で `space = "<spaceKey>"` かつ タイトルや本文に「タスクテンプレート」「テンプレート」「タスクの書き方」などを含むページを検索。  
   - または、スペースのホーム／目次ページの子（`confluence_get_page_children`）のうち、テンプレートと分かるページを取得。  
   - 複数候補がある場合は、Epic やタスク作成と関連が強いものを 1 つ選ぶ。
2. **テンプレートの構造を把握**  
   取得したページの見出し・表・箇条書きの項目名を確認し、「タスク 1 本あたりに含めるべき項目」として整理する。
3. **タスク内容の作成**  
   Epic の説明（入力から得た内容）を、**テンプレートの各項目に対応する形で**記入する。テンプレートにない項目は追加しない（スペースの慣行に合わせる）。  
   - **QA・機能テスト**: テンプレートに「機能テスト」「QA」などの項目があれば、そこに「必要」or「不要」を必ず選択して記入する。テンプレートに該当項目がなければ、タスク本文の末尾に「QA: 機能テスト [必要/不要]」を補足で明記する。  
   - 機能テスト要否の判断: 振る舞いが変わる変更 → 必要。ドキュメントのみ・設定のみ等 → 不要。迷ったら「必要」。

---

## Jira プロジェクトの課題フォームをテンプレートとする場合

対象が Jira のプロジェクト（例: DATA）で、チケットの「主要な詳細」に **依頼内容**・**依頼の背景** などの項目がある場合。

1. **項目の把握**  
   画面では「依頼内容」「依頼の背景」などとして表示されるが、Jira 上では**カスタムフィールド**であることが多い。テンプレート構造は「依頼の背景」→「依頼内容」の順で 1 本のタスクにまとめる。
2. **タスク内容の構成**  
   - **依頼の背景**: Epic の説明・経緯・参考情報。  
   - **依頼内容**: やること・成果物。末尾に「QA: 機能テスト [必要/不要]」を補足する。
3. **Jira でチケット作成する場合**  
   「Jira でチケット作成するときの注意（カスタムフィールド）」に従う。

---

## Jira でチケット作成するときの注意（カスタムフィールド）

`jira_create_issue` では `description` にだけ書くと、画面上の「依頼内容」「依頼の背景」が空のままになることがある（それらが別カスタムフィールドのため）。

1. **作成後すぐに update する**  
   `jira_create_issue` で summary と description を入れて作成したあと、**依頼内容・依頼の背景** がそのプロジェクトのカスタムフィールドなら、`jira_update_issue` でそれらを埋める。
2. **カスタムフィールド ID の調べ方**  
   `jira_search_fields` でキーワード「依頼」を検索する。  
   - 依頼内容: `customfield_10806`（Paragraph）  
   - 依頼の背景: `customfield_10809`（Paragraph）  
   上記は DATA プロジェクトで利用している ID。他プロジェクトでは検索で確認する。
3. **update のやり方**  
   `jira_update_issue` の `fields` に `customfield_10806`（依頼内容の本文）と `customfield_10809`（依頼の背景の本文）を渡す。description にも同じ内容を書いておくと、標準の説明欄にも残る。

## ワークフロー（チェックリスト）

実行時は以下を順に満たす。

- [ ] 入力を 4 パターンのどれかで判定（Slack / Confluence / Jira プロジェクト URL / 直接テキスト）
- [ ] パターンに応じて Slack MCP / Confluence MCP / テキスト から Epic 説明を取得
- [ ] 対象を確定する（Confluence の spaceKey または Jira の project_key）
- [ ] テンプレートを取得する
  - Confluence の場合: そのスペースのタスクテンプレートページを検索・取得
  - Jira の場合: そのプロジェクトの課題フォーム（依頼内容・依頼の背景など）をテンプレートとする
- [ ] 取得したテンプレートの項目に合わせてタスク内容を 1 本以上作成する
- [ ] 各タスクで QA・機能テスト要否を必ず選択（テンプレートにあればその項目に、なければ文末に補足）
- [ ] ユーザーにタスク本文を提示、または Jira MCP でチケット作成
- [ ] Jira で作成した場合、依頼内容・依頼の背景がカスタムフィールドなら update でそれらを埋める

## 入力例とパターン対応

| ユーザー入力例 | パターン |
|----------------|----------|
| `https://mycompany.slack.com/archives/C0123ABC/p1234567890123456` | A: Slack MCP で該当メッセージ/スレッドを取得。内容から Epic・スペースを特定 |
| 「PROJ スペースの Epic『新決済フロー』のページでチケット作って」 | B: Confluence MCP で Epic 取得。spaceKey=PROJ を対象スペースとしてテンプレート検索 |
| 「Epic: ログ基盤整備。スペースは DEV。目的: …」 | C: テキストを Epic 説明に。スペースが明記されていればそのスペースのテンプレートを取得 |
| Jira プロジェクト／ボード URL | D: プロジェクトキーを抽出し、課題フォームをテンプレートとしてタスク作成 |

---

## 流れの例（テンプレートに合わせる）

1. **Epic の内容を取得**（Slack / Confluence / テキストのいずれか）
2. **対象スペースを確定**（例: PROJ）
3. **そのスペースのタスクテンプレートを Confluence で検索**  
   例: `confluence_search` で `space = "PROJ"` かつ「タスクテンプレート」でヒットしたページを取得
4. **テンプレートの構造を確認**  
   例: 取得したページに見出し「目的」「やること」「受け入れ条件」「QA（機能テスト）」がある
5. **Epic の内容を、テンプレートの各項目に当てはめて記入**  
   テンプレートにない見出しは作らない。QA・機能テストはテンプレートにあればその項目に、なければ文末に「QA: 機能テスト [必要/不要]」を補足。

---

## テンプレートに合わせた記入例（仮のテンプレート構造の場合）

対象スペースのテンプレートが「目的」「やること」「受け入れ条件」「QA（機能テスト: 必要/不要）」だった場合の記入例。実際の見出し・項目名は**必ず対象スペースのタスクテンプレートに合わせる**。

**機能テスト「必要」の例:**

```markdown
## 目的
新決済基盤への移行のため、自サービスから決済完了 API を呼び出す実装を行う。

## やること
- 新決済 API のクライアント SDK を導入する
- 既存「注文確定」処理のうち、決済完了通知部分を新 API 呼び出しに差し替える
- エラー時リトライとログ仕様に従う

## 受け入れ条件
- [ ] 新 API のステージング環境での結合確認が完了している
- [ ] 単体テストが追加され、CI で通る

## QA（機能テスト）
必要。ステージングで「注文確定 → 決済完了」の E2E を 1 シナリオ実施すること。
```

**機能テスト「不要」の例:**

```markdown
## 目的
共通ログフォーマットへの参加のため、自サービス設計書をログ仕様に合わせて更新する。

## やること
- 既存設計書の「ログ・監視」セクションを共通仕様に合わせて更新
- フィールド名・必須項目の差分を一覧化し、開発チームに共有

## 受け入れ条件
- [ ] 設計書が Confluence に反映されている
- [ ] レビュー指摘が解消されている

## QA（機能テスト）
不要。ドキュメント変更のみのため動作確認は不要。
```

---

## 出力例（要約）

- **入力**: 「〇〇スペースの Epic『YYY』に参加するチケットを作って」
- **やること**: 対象スペースを特定 → Confluence でそのスペースのタスクテンプレートを取得 → テンプレートの形式でタスク内容を生成。QA・機能テスト要否を必ず選択して返す。
