# Name: tdd-refactor

---

# Description: TDDのRefactorフェーズを実行します。テストを維持しながらコードの品質を改善し、セキュリティ・パフォーマンスレビューも行います。

---

# Content

## 目的

TDDのRefactorフェーズを実行する。Greenフェーズで実装されたコードを改善し、**テストは必ず通り続けること**が大前提。

## 信頼性レベル指示

リファクタリング時には、各改善内容について元の資料との照合状況を以下の信号でコメントしてください：

- 🟢 **青信号**: 元の資料を参考にしてほぼ推測していない場合
- 🟡 **黄信号**: 元の資料から妥当な推測の場合
- 🔴 **赤信号**: 元の資料にない推測の場合

## 改善の観点

### 1. 可読性の向上

- 変数名・関数名の改善
- 日本語コメントの充実
- コードの構造を分かりやすく

### 2. 重複コードの除去（DRY原則）

- 同じような処理の共通化
- 定数の抽出
- ヘルパー関数の作成

### 3. 設計の改善

- 単一責任原則の適用
- 依存関係の整理
- モジュール化の検討

### 4. ファイルサイズの最適化

- ファイルサイズが適切になるよう分割・最適化
- 長大なファイルの機能別分割
- 適切なモジュール境界の設定

### 5. コード品質の確保

- lintエラーの解消
- typecheckエラーの解消
- フォーマットの統一
- 静的解析ツールのチェッククリア

### 6. セキュリティレビュー

- 脆弱性に繋がる実装の検出と修正
- 入力値検証の強化
- SQLインジェクション対策の確認
- XSS（Cross-Site Scripting）対策の確認
- データ漏洩リスクの回避
- 認証・認可の適切な実装

### 7. パフォーマンスレビュー

- アルゴリズムの計算量解析
- メモリ使用量の最適化
- 不要な処理の削除
- キャッシュ戦略の検討
- データベースクエリの最適化
- ループ処理の効率化

### 8. エラーハンドリングの充実

- 入力値の検証
- 適切なエラーメッセージ
- 例外処理の改善

## リファクタリングの手順

1. **現在のテストが全て通ることを確認**
   - リファクタ前の動作確認
   - 変更による機能破綻の防止

2. **コード・テスト除外チェック**
   - 本来確認対象のコードファイルが除外されていないかチェック
   - テストが無効化されていないかチェック
   - 実際に実行されるべきテストとコードが適切に対象に含まれているか確認

3. **不要ファイルのクリーンアップ**
   - 開発中に作成された一時的なファイルを検出・削除
   - 削除前に各ファイルの内容を確認

4. **セキュリティレビューの実施**
   - コード全体のセキュリティホールの特定
   - 入力検証確認

5. **パフォーマンスレビューの実施**
   - アルゴリズムの計算量解析
   - ボトルネック特定

6. **小さな改善を1つずつ適用**
   - 影響範囲を限定した安全な変更
   - 変更内容の追跡可能性確保

7. **各改善後にテストを実行**
   - 改善の度に動作確認
   - 問題の早期発見と修正

8. **テストが失敗したら即座に戻す**
   - 問題発生時の素早い対応
   - システムの安定した状態を保持

## 注意事項

- **機能的な変更は行わない**（新機能追加はNG）
- **テストが通らなくなったら即座に修正**
- **一度に大きな変更をしない**
- **日本語コメントの品質も向上させる**

## 提供してください

1. **セキュリティレビュー結果**: 脆弱性の有無と対応策
2. **パフォーマンスレビュー結果**: 性能課題の分析と改善策
3. **改善されたコード**: リファクタリング後のコード（強化された日本語コメント付き）
4. **改善ポイントの説明**: 何をどのように改善したか
5. **テスト実行結果**: 全てのテストが引き続き通ることの確認
6. **品質評価**: 現在のコードの品質レベル

## 実行後の処理

リファクタリング完了後、以下を実行してください：

1. **メモファイル最終更新**: `docs/implements/{task_id}/{feature_name}-memo.md`ファイルのRefactorフェーズセクションを更新
   - 改善内容、セキュリティレビュー結果、パフォーマンスレビュー結果を記録
2. リファクタリング内容を`docs/implements/{task_id}/{feature_name}-refactor-phase.md`に保存
3. **品質判定**: リファクタリング成果の品質を以下の基準で判定

## 品質判定基準

```
✅ 高品質:
- テスト結果: 全て継続成功
- セキュリティ: 重大な脆弱性なし
- パフォーマンス: 重大な性能課題なし
- リファクタ品質: 目標達成
- コード品質: 適切なレベル

⚠️ 要改善:
- テストの一部失敗
- セキュリティ脆弱性発見
- パフォーマンス課題発見
- リファクタ目標未達成
- 品質改善不十分
```

## 出力フォーマット

```

### セキュリティレビュー結果
- [セキュリティ上の懸念点と対応]

### パフォーマンスレビュー結果
- [パフォーマンス上の懸念点と対応]

### 変更内容
1. [変更1の説明]
2. [変更2の説明]

### 変更前
[変更前のコード]

### 変更後
[変更後のコード]

### テスト実行結果
[テスト実行結果（PASS）]

✅ リファクタリング完了、すべてのテストがパスしています
```

### リファクタリング不要の場合

```
## 🔵 Refactor: 確認完了

現在のコードを確認しました。

### 評価結果
- 重複コード: なし
- 命名: 適切
- 複雑度: 低
- セキュリティ: 問題なし
- パフォーマンス: 問題なし

リファクタリングの必要はありません。
```
