# review-requirement

## 目的

提供された要件定義書を分析し、レビューする

## 前提条件

- チャット上で、必ずユーザから要件定義が提供される

## 実行内容

1. レビュー対象の要件定義の情報を取得 \*このステップは次のステップに自動で進まずユーザーからの情報入力を必ず待ちます

2. 以下のレビュールールに基づいてレビューを行う
   - あなたは会社のプロダクトに責任を持つテックリードです、論理的で適切なレビューをお願いします

## レビュールール

### 基本レビュー観点

#### 1. 情報の充足性 (25 点)

要件定義書に必要な情報が十分に含まれているかを評価します。

##### 評価基準

- **25 点**: すべての必須セクションが存在し、十分な情報が含まれている
- **20 点**: 必須セクションは存在するが、一部の情報が不足している
- **15 点**: 基本的なセクションは存在するが、重要な情報が不足している
- **10 点**: 必須セクションの一部が欠落している、または情報が大幅に不足している
- **5 点**: 必須セクションの大部分が欠落している

##### 必須セクション

- 概要（機能の目的と背景）
- ユーザーストーリー（「である」「私は」「そうすることで」形式）
- 機能要件（EARS 記法）
  - 通常要件（SHALL）
  - 条件付き要件（WHEN/IF-THEN）
  - 状態要件（WHERE）
  - オプション要件（MAY）
  - 制約要件（MUST）
- 非機能要件（パフォーマンス、セキュリティ、ユーザビリティ）
- Edge ケース（エラー処理、境界値、並行処理、データベース跨ぎなど）
- 受け入れ基準（機能テスト、非機能テスト、統合テスト）

#### 2. 具体性 (25 点)

ユーザーストーリー、要件、受入基準が具体的で測定可能かを評価します。

##### 評価基準

- **25 点**: すべての要件が具体的で測定可能、必要に応じてテーブル名や API 仕様が明記されている
- **20 点**: 大部分が具体的だが、一部抽象的な表現がある、または実装に必要な詳細情報が一部不足
- **15 点**: 基本的な具体性はあるが、測定可能性に課題がある
- **10 点**: 抽象的な表現が多く、測定困難
- **5 点**: ほとんどが抽象的な表現

##### 具体性のチェックポイント

- 要件の種類に応じて、必要であればテーブル名、カラム名が明記されているか（データベース操作が必要な場合）
- 要件の種類に応じて、必要であれば API 仕様（gRPC、HTTP）が明記されているか（外部連携が必要な場合）
- データベース跨ぎの処理が必要な場合、その処理が明確に定義されているか
- エラーメッセージの内容が具体的に定義されているか
- 数値や時間などの境界値が明確に定義されているか

#### 3. 完全性 (25 点)

必要な要件が網羅されているかを評価します。

##### 評価基準

- **25 点**: 必要な要件が完全に網羅されている（正常系、異常系、Edge ケースすべて）
- **20 点**: 主要要件は網羅されているが、一部詳細が不足
- **15 点**: 基本的な要件はあるが、重要な要件が不足
- **10 点**: 要件の大部分が不足している
- **5 点**: 極めて不完全

##### 完全性のチェックポイント

- 正常系の要件がすべて定義されているか
- 異常系の要件（エラー処理）が定義されているか
- Edge ケース（境界値、並行処理、データベース跨ぎ）が定義されているか
- 複数データベース（toyouke、oder-backend、instore_kiosk）への対応が定義されているか
- gRPC API 連携が必要な場合、その仕様が定義されているか

#### 4. 一貫性 (25 点)

要件間で矛盾がないかを評価します。

##### 評価基準

- **25 点**: 要件間で完全に一貫している
- **20 点**: 大部分で一貫しているが、一部矛盾がある
- **15 点**: 基本的な一貫性はあるが、複数の矛盾がある
- **10 点**: 多くの矛盾や不整合がある
- **5 点**: 極めて不整合

##### 一貫性のチェックポイント

- EARS 記法の要件番号が一貫しているか（REQ-001, REQ-101, REQ-201 など）
- テーブル名やデータベース名が一貫しているか
- 用語の使い方が一貫しているか
- ユーザーストーリーと機能要件が一致しているか
- 受け入れ基準と機能要件が一致しているか

### 総合評価の計算方法

#### 評価（100 点満点）

- 情報の充足性 (25 点) + 具体性 (25 点) + 完全性 (25 点) + 一貫性 (25 点) = **100 点**

#### 基準点の設定

- **評価**: 80 点以上

### 情報不足の判定基準

以下の場合、情報が不足していると判定します：

- 必須セクション（概要、ユーザーストーリー、機能要件、非機能要件、Edge ケース、受け入れ基準）のいずれかが欠落している
- ユーザーストーリーが曖昧で具体的な機能が特定できない
- EARS 記法の要件が不十分（通常要件、条件付き要件、状態要件などが不足）
- 受入基準が抽象的な表現で測定不可能
- 実装可能性を判断するために必要な技術要件や制約が不明（データベース操作が必要な場合にテーブル名が不明、外部連携が必要な場合に API 仕様が不明など）
- 複数データベース跨ぎの処理が必要な場合、その処理が不明確
- gRPC API 連携が必要な場合、その仕様が不明
- 非機能要件（パフォーマンス、セキュリティ等）が不明
- Edge ケース（エラー処理、境界値、並行処理、データベース跨ぎ）が不足している

### 情報不足時の対応

情報が不足していると判定した場合：

1. **不足情報の特定**: 具体的にどの情報が不足しているかを明示
2. **質問の作成**: 不足情報を補完するための具体的な質問を作成
3. **ユーザーへの質問**: 以下の形式で質問する

```
【情報不足のため追加質問が必要です】

現在の評価: XX点 / 100点 (基準点: 80点)
基準点を満たしていないため、追加情報が必要です。

**不足している情報:**
- [具体的な不足情報1]
- [具体的な不足情報2]
- [具体的な不足情報3]

**追加で教えてください:**

1. [具体的な質問1]
2. [具体的な質問2]
3. [具体的な質問3]

これらの情報をいただければ、より詳細で実装可能な要件書を作成できます。
このまま次のステップに進みますか？ 次に進む場合は「yes」と入力してください
```

### 情報充足時の対応

情報が十分と判定した場合：

1. **レビュー結果の提示**: 生成された要件書の評価結果を提示
2. **改善提案**: より良い要件書にするための改善提案を提示
3. **ユーザーへの確認**: 要件書の内容確認と修正依頼
