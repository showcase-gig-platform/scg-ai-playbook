# Content

## 目的

承認された要件定義書に基づいて、技術設計文書を生成する。データフロー図、型/インターフェース定義、データベーススキーマ、APIエンドポイントを含む包括的な設計を行う。

## 前提条件

- `docs/spec/` に要件定義書が存在する
- 要件がユーザによって承認されている

## 実行内容

1. **要件の分析**
   - 要件定義書を読み込む
   - 機能要件と非機能要件を整理する
   - システムの境界を明確にする

2. **アーキテクチャ設計**
   - システム全体のアーキテクチャを決定
   - レイヤー構成を定義
   - コンポーネント間の依存関係を設計

3. **データフロー図の作成**
   - Mermaid記法でデータフローを可視化
   - ユーザーインタラクションの流れ
   - システム間のデータの流れ

4. **型/インターフェースの定義**
   - エンティティの型定義
   - APIリクエスト/レスポンスの型定義
   - 共通型の定義

5. **データベーススキーマの設計**
   - テーブル定義
   - リレーションシップ
   - インデックス戦略
   - 正規化レベルの決定

6. **APIエンドポイントの設計**
   - RESTful API / gRPC / GraphQL設計
   - エンドポイントの命名規則
   - リクエスト/レスポンスの構造

7. **ファイルの作成**
   - `docs/design/{要件名}/` ディレクトリに以下を作成：
     - `architecture.md` - アーキテクチャ概要
     - `dataflow.md` - データフロー図
     - `interfaces.[ext]` - 型定義（言語に応じた拡張子）
     - `database-schema.sql` - DBスキーマ
     - `api-endpoints.md` - API仕様

## 言語別の型定義ファイル

プロジェクトの言語に応じて、型定義ファイルを作成してください：

| 言語 | ファイル名 | 形式 |
|------|-----------|------|
| Go | `interfaces.go` | struct定義 |
| Rust | `interfaces.rs` | struct/enum定義 |
| Python | `interfaces.py` | dataclass/Pydantic |
| TypeScript | `interfaces.ts` | interface/type定義 |
| Java | `interfaces.java` | class/interface定義 |

## 出力フォーマット例

### architecture.md

```markdown
# {要件名} アーキテクチャ設計

## システム概要
{システムの概要説明}

## アーキテクチャパターン
- パターン: {選択したパターン}
- 理由: {選択理由}

## コンポーネント構成

### プレゼンテーション層
- フレームワーク: {使用フレームワーク}
- 状態管理: {状態管理方法}

### ビジネスロジック層
- 構成: {サービス/ユースケース構成}

### データアクセス層
- ORM/ドライバ: {使用するORM/ドライバ}

### インフラストラクチャ層
- データベース: {使用するDBMS}
- キャッシュ: {キャッシュ戦略}
```

### dataflow.md

```markdown
# データフロー図

## ユーザーインタラクションフロー

\`\`\`mermaid
flowchart TD
    A[ユーザー] --> B[クライアント]
    B --> C[API Gateway]
    C --> D[ビジネスロジック]
    D --> E[データベース]
\`\`\`

## データ処理フロー

\`\`\`mermaid
sequenceDiagram
    participant U as ユーザー
    participant C as クライアント
    participant S as サーバー
    participant D as データベース
    
    U->>C: アクション
    C->>S: リクエスト
    S->>D: クエリ実行
    D-->>S: 結果返却
    S-->>C: レスポンス
    C-->>U: 画面更新
\`\`\`
```
